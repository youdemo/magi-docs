## 1. 对话流程系统分析

### 1.1 设计理念（Message Protocol）

**核心设计** (`src/protocol/message-protocol.ts`):
- 统一消息协议 (Unified Message Protocol)
- 明确的消息类型枚举 (10种类型)
- 清晰的生命周期状态机 (5个状态)
- 结构化内容块 (5种block类型)

```typescript
// 消息生命周期
enum MessageLifecycle {
  STARTED     → STREAMING → COMPLETED  // 正常流程
  STARTED     → STREAMING → FAILED     // 错误流程
  STARTED     → STREAMING → INTERRUPTED // 中断流程
}
```

**设计优点**:
1. ✅ 类型安全，消除格式猜测
2. ✅ 扩展性强，支持富文本元素
3. ✅ 标准化，Webview只负责渲染

### 1.2 实现现状（代码为准）

**消息流路径（双通道并存）**:
```
通道 A: 标准消息（StandardMessage）
CLI Adapter output
  ↓ (CLIAdapterFactory -> Normalizer)
standardMessage / standardUpdate / standardComplete
  ↓ (WebviewProvider 去重后 postMessage)
Webview (standardMessage 渲染)

通道 B: Legacy 流式输出
CLIAdapterFactory.emit('output'/'response') / globalEventBus('cli:output')
  ↓ (WebviewProvider sendStreamEvent)
Webview (stream 片段渲染)
```

**关键事实**:
1. `MessageBridge` 存在但没有被使用；WebviewProvider 直接监听 `CLIAdapterFactory` 的标准事件。
2. WebviewProvider 同时监听 `standardMessage` 与 `output/response`/`cli:output` 事件。
3. 去重器仅应用于标准消息通道，Legacy 流式通道不做去重。

**问题诊断**:

#### 问题1: 标准消息与 Legacy 流双通道并存 (P1)

**现象**: 同一任务存在两套渲染链路，标准消息与流式片段可能在 UI 上并行出现，去重规则只覆盖标准通道。

**代码证据**:
1. `src/ui/webview-provider.ts:217-427` 同时监听标准消息与 `output/response`。
2. `src/cli/adapter-factory.ts:419-435` 编排者摘要仍走 `emit('output')`。
3. `src/ui/webview-provider.ts:834-843` 仍监听 `globalEventBus('cli:output')` 直推流式输出。

**影响范围**:
- Thread 面板：编排者消息既可能走标准消息，也可能走 Legacy stream。
- CLI 面板：`cli:output` 与标准消息并存，去重仅对标准消息有效。
- 复杂情况下可能出现“重复/覆盖/顺序错乱”的表现。

**修复建议**:
```typescript
// 1. 统一通道：逐步移除 WebviewProvider 对 output/response 的监听
// 2. 编排者消息统一走 standardMessage（或明确只走 Legacy，不双写）
// 3. 若保留双通道，需为 Legacy stream 引入去重与生命周期校验
```

#### 问题2: 状态同步已接入但缺少统一模型 (P2)

**现状**:
- 编排者状态通过 `orchestrator:phase_changed` 推送到 UI。
- 消息生命周期由标准消息与 Legacy stream 分别驱动。
- 两者之间没有统一状态模型或一致性校验。

**风险**:
- UI 层只能基于 `phaseChanged` 和单条消息更新推断运行状态。
- 缺少统一模型时，阶段展示与消息流可能在边界情况出现不一致。

**建议**:
- 引入统一状态模型（编排者状态 + 消息生命周期 + UI 阶段），统一发布。

## 2. 编排系统业务流程分析

### 2.1 设计理念（Orchestrator Architecture）

**独立编排者架构** (`orchestrator-agent.ts:3-18`):
- 编排者 Claude: 专职编排，不执行编码
- Worker Agents: 专职执行，向编排者汇报
- 事件循环: 实时监控所有 Worker
- Intent Gate: 意图门控，防止过度实施

**核心流程**:
```
用户输入
  ↓
Phase 0: Intent Gate (意图分类)
  ↓
Phase 0.5: 需求澄清 (如需要)
  ↓
Phase 1: 任务分析 (生成计划)
  ↓
Phase 2: 计划确认 (用户审批)
  ↓
Phase 3: 任务执行 (Worker 并行/串行)
  ↓
Phase 4: 验证 (编译/测试)
  ↓
Phase 5: 集成整理 (生成总结)
```

### 2.2 实现现状与问题

#### 问题3: Intent Gate 执行时机偏晚 (P2)

**当前实现（代码为准）**:
```typescript
// src/orchestrator/orchestrator-agent.ts:912-930（附近）
// 先 ensureContext，再执行 Intent Gate
await this.ensureContext(contextSessionId, userPrompt);
const intentResult = this.intentGate.process(userPrompt);
```

**问题根源**:
- Intent Gate 在 `ensureContext()` 之后执行。
- QUESTION/EXPLORE 类轻量请求仍会初始化上下文、会话、快照等组件。

**影响**:
- 轻量请求仍承担上下文初始化开销（延迟 + 资源消耗）。

**修复建议**:
```typescript
// 1️⃣ FIRST: Intent Gate
const intentResult = this.intentGate.process(userPrompt);

// 2️⃣ 轻量意图直接返回
if (intentResult.classification.type === 'QUESTION') {
  return await this.handleQuestionDirectly(userPrompt);
}
if (intentResult.classification.type === 'EXPLORE') {
  return await this.handleExploreDirectly(userPrompt);
}

// 3️⃣ 仅对 TASK 类意图初始化上下文
await this.ensureContext(contextSessionId, userPrompt);
```
```

## 3. 任务系统与依赖管理分析

### 3.1 设计理念（Task Dependency Graph）

**核心机制** (`task-dependency-graph.ts`):
- DAG 调度: 有向无环图管理任务依赖
- 文件依赖检测: 自动识别文件冲突
- 死锁检测: 防止循环依赖

**依赖类型**:
1. 显式依赖: 用户声明的 `dependencies: ['task-1']`
2. 文件依赖: 两个任务修改同一文件 → 自动串行化

### 3.2 实现现状与问题

#### 问题5: 文件冲突检测未接入调度图 (P1 高)

**当前架构（代码为准）**:
```
TaskDependencyGraph (支持 targetFiles 冲突检测)
     ↓
WorkerPool 构建图时未传入 targetFiles
     ↓
文件冲突检测与 addFileDependencies 未执行
     ↓
执行期仅依赖 FileLockManager（基于 targetFiles 或正则推断）
```

**代码证据**:
- `TaskDependencyGraph` 支持 `targetFiles` 与 `detectFileConflicts/addFileDependencies`。
- `WorkerPool.executeWithDependencyGraph()` / `createDependencyGraph()` 构建图时未传入 `targetFiles`。
- FileLockManager 在执行期锁文件，但锁集合来自 `subTask.targetFiles` 或正则推断。

**风险**:
- 当 `targetFiles` 缺失或推断不准时，写冲突无法在计划阶段被串行化。
- SnapshotManager 发现冲突仅告警，不阻塞执行（见 `src/snapshot-manager.ts:78-115` 附近）。

**修复建议**:
1. WorkerPool 构建依赖图时传入 `subTask.targetFiles`。
2. 在图分析阶段执行 `addFileDependencies()` 并输出冲突分析。
3. 对无法确定目标文件的子任务进行保守串行化或强制确认。

**优先级**: P1 - 高
**影响**: 数据完整性、回滚可靠性

## 4. 快照系统分析

### 4.1 设计理念

**核心功能** (`snapshot-manager.ts`):
- 创建快照: 任务执行前保存原始文件
- 还原快照: 回滚到任务前状态
- 变更追踪: 对比原始与当前文件
- 批量操作: 接受/还原所有变更

**存储结构**:
```
.multicli/sessions/{sessionId}/snapshots/
  ├── {snapshotId-1}.snapshot
  ├── {snapshotId-2}.snapshot
  └── ...
```

### 4.2 实现现状与问题

#### 问题6: 快照粒度过粗且无历史 (P2)

**当前实现**:
```typescript
// 每个SubTask一个快照
createSnapshot(filePath, modifiedBy, subTaskId)
```

**问题场景**:
```
SubTask-1: "重构 types.ts 和 utils.ts"
  → 创建 types.ts snapshot
  → 创建 utils.ts snapshot

如果只想回滚 types.ts？无法单独回滚！
```

**缺失功能**:
1. ❌ 无法部分回滚（只能按文件快照回滚）
2. ❌ 无变更预览（回滚前看 diff）
3. ❌ 无快照历史（同一文件快照会被覆盖）

**代码证据**:
- `UnifiedSessionManager.addSnapshot()` 对相同 `filePath` 直接覆盖元数据（仅保留最新快照）。

## 5. 上下文系统与记忆管理分析

### 5.1 设计理念（三层上下文）

**架构设计** (`context-manager.ts:2-10`):
```
Layer 1: 即时上下文 (Immediate Context)
  - 最近几轮对话
  - Augment风格截断
  - 保持在 token 限制内

Layer 2: 会话Memory (Session Memory)
  - 结构化任务记录
  - 关键决策
  - 代码变更

Layer 3: 项目知识库 (未实现)
  - 跨会话知识
  - 项目约定
  - 历史问题
```

### 5.2 实现现状与问题

#### 问题7: 即时上下文与 Memory 无自动迁移 (P1)

**当前实现（代码为准）**:
```typescript
// src/context/context-manager.ts:69-100
// 即时上下文直接截断并裁剪，超出上限直接丢弃
this.immediateContext.push(msg);
// …超过 maxMessages 后，仅保留 system + 最近若干条
```

**断层表现**:
1. 即时上下文达到上限后直接裁剪，无自动迁移到 Memory。
2. Memory 只通过显式调用写入（如 addCurrentTask/addDecision/addImportantContext）。
3. 长会话中早期用户输入可能丢失，除非显式写入 Memory。

**建议**:
- 在 addMessage 时对关键消息自动迁移到 Memory（用户输入、关键结论、错误）。
- 超限裁剪时将被移除内容以摘要方式写入 Memory。

#### 问题8: 压缩已接入，但 LLM 适配器未设置 (P2)

**当前实现（代码为准）**:
- `ContextCompressor` 存在并在任务结束时触发（`saveAndCompressMemory()`）。
- 但未设置 LLM 适配器，实际只会走截断 + 简单压缩路径。

**风险**:
- Memory 规模增长时，压缩效果有限，仍可能丢失关键信息。

**建议**:
- 为 `ContextCompressor` 注入可用的 LLM Adapter；
- 增加压缩质量度量（保留关键决策、当前任务不丢失）。

## 6. 系统性风险总结

### 6.1 数据完整性风险 (P1)

**风险**: 文件冲突检测未接入计划阶段 → 执行期仅依赖 FileLockManager（可能缺少完整文件列表）

**触发条件**:
1. `targetFiles` 缺失或推断失败
2. 多个 SubTask 实际修改同一文件
3. 快照冲突仅告警、不阻塞

**后果**:
- 文件内容被覆盖或顺序错误
- 快照回滚效果不可靠

**修复优先级**: P1 - 高

### 6.2 用户体验风险 (P1)

**风险1**: 标准消息与 Legacy stream 并存 → UI 可能重复/覆盖/乱序
**风险2**: 状态与消息生命周期未统一 → 边界状态展示不稳定
**风险3**: 长会话上下文丢失 → 早期信息被截断丢失

**修复优先级**: P1 - 本周内修复

### 6.3 性能风险 (P2)

**风险**: Intent Gate 在 `ensureContext()` 之后执行 → 轻量请求仍初始化完整上下文

**场景**: 用户问“什么是 TypeScript?” 仍会初始化会话与上下文

**影响**: 延迟与资源浪费

**修复优先级**: P2 - 下周修复
