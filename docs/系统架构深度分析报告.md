## 1. 对话流程系统分析

### 1.1 设计理念（Message Protocol）

**核心设计** (`src/protocol/message-protocol.ts`):
- 统一消息协议 (Unified Message Protocol)
- 明确的消息类型枚举 (10种类型)
- 清晰的生命周期状态机 (5个状态)
- 结构化内容块 (5种block类型)

```typescript
// 消息生命周期
enum MessageLifecycle {
  STARTED     → STREAMING → COMPLETED  // 正常流程
  STARTED     → STREAMING → FAILED     // 错误流程
  STARTED     → STREAMING → INTERRUPTED // 中断流程
}
```

**设计优点**:
1. ✅ 类型安全，消除格式猜测
2. ✅ 扩展性强，支持富文本元素
3. ✅ 标准化，Webview只负责渲染

### 1.2 实现现状

**消息流路径** (4层转换):
```
CLI原始输出
  ↓
Normalizer (标准化)
  ↓
MessageBridge (生命周期转换)
  ↓
MessageDeduplicator (去重)
  ↓
WebviewProvider (UI渲染)
```

**问题诊断**:

#### 问题1: 消息重复发送 (P1)

**现象**: 对话面板出现消息覆盖，进度消息合并不当

**根本原因**:
1. **双重事件流**:
   - `adapter-factory.ts:177-179`: 旧的 `output` 事件被注释但未彻底移除
   - `adapter-factory.ts:108-109`: 新的 `standardMessage` 事件正常发送
   - WebviewProvider 同时监听两种事件，导致消息重复

2. **去重器配置**:
   ```typescript
   // message-deduplicator.ts:48
   minStreamInterval: 100,  // 默认100ms - 太短
   enabled: true,           // 总是启用 - 不够灵活
   ```

3. **生命周期转换混乱**:
   ```typescript
   // message-bridge.ts 缺少状态机验证
   // STARTED → COMPLETED 直接跳转，跳过 STREAMING
   ```

**影响范围**:
- Thread面板: 编排者消息可能重复
- CLI面板: Worker消息可能重复
- 用户体验: 消息闪烁、覆盖、顺序混乱

**修复建议**:
```typescript
// 1. 彻底移除旧API事件
// adapter-factory.ts: 删除 output/response 事件监听

// 2. 调整去重器参数
const DEFAULT_CONFIG = {
  minStreamInterval: 300,  // 增加到300ms
  enabled: true,
  maxMessagesPerSource: 1000,
  cleanupInterval: 60000,
}

// 3. 强制生命周期顺序校验
private validateLifecycleTransition(
  from: MessageLifecycle,
  to: MessageLifecycle
): boolean {
  const validTransitions = {
    [MessageLifecycle.STARTED]: [MessageLifecycle.STREAMING],
    [MessageLifecycle.STREAMING]: [MessageLifecycle.COMPLETED, MessageLifecycle.FAILED],
  };
  return validTransitions[from]?.includes(to) ?? false;
}
```

#### 问题2: 对话流程状态不同步 (P1)

**多层状态机现状**:
- `OrchestratorAgent._state`: 12个状态 (analyzing, planning, executing, ...)
- `MessageLifecycle`: 5个状态 (started, streaming, ...)
- `WebviewProvider.phase`: UI显示的阶段

**冲突场景**:
```typescript
// 场景1: 编排者状态 = analyzing，但UI显示 = executing
// orchestrator-agent.ts:948
this.setState('analyzing');  // 更新内部状态
plan = await this.analyzeTask(prompt);  // 长时间执行

// 但WebviewProvider未收到 analyzing 事件
// 因为 setupOrchestratorEvents() 只转发 stateChange，不转发消息状态
```

**修复建议**:
```typescript
// 统一状态模型
interface UnifiedTaskState {
  orchestratorPhase: OrchestratorState;   // 编排者阶段
  messageLifecycle: MessageLifecycle;     // 消息生命周期
  uiDisplayPhase: string;                 // UI显示阶段
  timestamp: number;
}

// 状态同步中心
class StateSynchronizer {
  private state: UnifiedTaskState;

  updateOrchestratorPhase(phase: OrchestratorState) {
    this.state.orchestratorPhase = phase;
    this.state.uiDisplayPhase = this.mapToUIPhase(phase);
    this.emit('stateChanged', this.state);
  }

  private mapToUIPhase(phase: OrchestratorState): string {
    // analyzing → 分析中
    // planning → 计划中
    // executing → 执行中
  }
}
```

## 2. 编排系统业务流程分析

### 2.1 设计理念（Orchestrator Architecture）

**独立编排者架构** (`orchestrator-agent.ts:3-18`):
- 编排者 Claude: 专职编排，不执行编码
- Worker Agents: 专职执行，向编排者汇报
- 事件循环: 实时监控所有 Worker
- Intent Gate: 意图门控，防止过度实施

**核心流程**:
```
用户输入
  ↓
Phase 0: Intent Gate (意图分类)
  ↓
Phase 0.5: 需求澄清 (如需要)
  ↓
Phase 1: 任务分析 (生成计划)
  ↓
Phase 2: 计划确认 (用户审批)
  ↓
Phase 3: 任务执行 (Worker 并行/串行)
  ↓
Phase 4: 验证 (编译/测试)
  ↓
Phase 5: 集成整理 (生成总结)
```

### 2.2 实现现状与问题

#### 问题3: Intent Gate 执行时机不当 (P2)

**当前实现**:
```typescript
// orchestrator-agent.ts:916-930
async execute(userPrompt, taskId, sessionId) {
  // 1. 初始化任务上下文
  this.currentContext = { ... };
  this.abortController = new AbortController();

  // 2. 初始化上下文管理器
  await this.ensureContext(contextSessionId, userPrompt);

  // 3. Intent Gate处理
  const intentResult = this.intentGate.process(userPrompt);

  // 问题：在上下文加载后才分类意图
}
```

**问题根源**:
- Intent Gate 应该是**第一步**，而非第三步
- 原因：QUESTION/EXPLORE 类意图不需要完整上下文
- 当前设计：所有请求都加载上下文 → 浪费资源

**影响**:
- QUESTION类请求（"什么是X？"）也初始化完整任务上下文
- EXPLORE类请求（"列出所有文件"）也创建会话和快照管理器
- 上下文初始化耗时 100-300ms，无谓延迟

**修复建议**:
```typescript
async execute(userPrompt, taskId, sessionId) {
  // 1️⃣ FIRST: Intent Gate (最轻量)
  const intentResult = this.intentGate.process(userPrompt);

  // 2️⃣ 如果是轻量意图，立即处理并返回
  if (intentResult.classification.type === 'QUESTION') {
    return await this.handleQuestionDirectly(userPrompt);
  }
  if (intentResult.classification.type === 'EXPLORE') {
    return await this.handleExploreDirectly(userPrompt);
  }

  // 3️⃣ ONLY THEN: 初始化完整上下文（仅对TASK类）
  this.currentContext = { ... };
  await this.ensureContext(contextSessionId, userPrompt);

  // 4️⃣ 继续任务分析流程
  const plan = await this.analyzeTask(userPrompt);
  // ...
}
```

## 3. 任务系统与依赖管理分析

### 3.1 设计理念（Task Dependency Graph）

**核心机制** (`task-dependency-graph.ts`):
- DAG 调度: 有向无环图管理任务依赖
- 文件依赖检测: 自动识别文件冲突
- 死锁检测: 防止循环依赖

**依赖类型**:
1. 显式依赖: 用户声明的 `dependencies: ['task-1']`
2. 文件依赖: 两个任务修改同一文件 → 自动串行化

### 3.2 实现现状与问题

#### 问题5: 快照系统与文件锁不协调 (P0 严重)

**当前架构**:
```
TaskDependencyGraph (任务调度)
     ↓
  并行执行 task-1, task-2
     ↓
SnapshotManager (快照创建)
     ↓
  发现冲突，但已经太晚！
```

**问题场景**:
```typescript
// snapshot-manager.ts:86-100
if (otherSnapshot) {
  // ⚠️ WARNING: 多个SubTask修改同一文件
  console.warn('Multiple SubTasks modifying same file');
  // 但只是警告，继续创建新快照！
}
```

**根本原因**:
- **检测时机错误**: 快照在任务**执行时**创建，而非**计划时**检测
- **无阻塞机制**: 发现冲突后只警告，不阻止任务执行
- **数据竞争**: 两个Worker同时写入同一文件 → 数据损坏

**实际风险案例**:
```
时间线:
T0: task-1 (Codex) 开始修改 src/types.ts
T1: task-2 (Gemini) 并行开始，也修改 src/types.ts
T2: task-1 完成，写入types.ts (版本A)
T3: task-2 完成，写入types.ts (版本B) → 覆盖版本A
T4: 用户回滚，快照混乱，数据无法恢复
```

**修复方案**:

```typescript
// 方案1: 提前检测并强制串行化
class TaskDependencyGraph {
  async addTask(task: SubTaskDef) {
    // 1. 检查文件冲突
    for (const file of task.targetFiles) {
      const existingTask = this.fileOwnerMap.get(file);
      if (existingTask) {
        // 2. 自动添加依赖
        task.dependencies.push(existingTask.id);
        console.log(`[FileLock] ${task.id} 等待 ${existingTask.id} 完成（文件冲突: ${file}）`);
      }
    }

    // 3. 死锁检测
    if (this.detectCycle(task)) {
      throw new Error(`[FileLock] 检测到循环依赖: ${task.id}`);
    }
  }
}

// 方案2: 文件锁管理器集成
class IntegratedFileLockManager {
  private locks: Map<string, {
    owner: string;  // 任务ID
    mode: 'read' | 'write';
    acquiredAt: number;
  }> = new Map();

  async acquireLock(taskId: string, files: string[], mode: 'read' | 'write'): Promise<void> {
    for (const file of files) {
      const lock = this.locks.get(file);
      if (lock && lock.mode === 'write') {
        // 等待写锁释放
        await this.waitForRelease(file);
      }
    }

    // 获取锁
    for (const file of files) {
      this.locks.set(file, { owner: taskId, mode, acquiredAt: Date.now() });
    }
  }

  releaseLock(taskId: string, files: string[]): void {
    for (const file of files) {
      const lock = this.locks.get(file);
      if (lock?.owner === taskId) {
        this.locks.delete(file);
      }
    }
  }
}
```

**优先级**: P0 - 必须立即修复
**影响**: 数据完整性、用户信任度

## 5. 上下文系统与记忆管理分析

### 5.1 设计理念（三层上下文）

**架构设计** (`context-manager.ts:2-10`):
```
Layer 1: 即时上下文 (Immediate Context)
  - 最近几轮对话
  - Augment风格截断
  - 保持在 token 限制内

Layer 2: 会话Memory (Session Memory)
  - 结构化任务记录
  - 关键决策
  - 代码变更

Layer 3: 项目知识库 (未实现)
  - 跨会话知识
  - 项目约定
  - 历史问题
```

### 5.2 实现现状与问题

#### 问题7: 三层上下文未集成 (P1)

**当前实现**:
```typescript
// 即时上下文
this.immediateContext.push(message);

// Memory
this.sessionMemory?.addCurrentTask(task);

// ❌ 两者独立，无自动同步
```

**断层表现**:
1. 即时上下文到达限制 → 旧消息被丢弃
2. 但这些消息未迁移到 Memory
3. 结果：长会话中早期信息丢失

**修复建议**:
```typescript
class IntegratedContextManager {
  addMessage(message) {
    // 1. 添加到即时上下文
    this.immediateContext.push(message);

    // 2. 检查是否需要迁移到Memory
    if (this.needsMigration(message)) {
      this.migrateToMemory(message);
    }

    // 3. 清理旧消息
    if (this.immediateContext.length > this.maxMessages) {
      const removed = this.immediateContext.shift();
      this.ensureMigratedToMemory(removed);
    }
  }

  private needsMigration(message): boolean {
    // 重要消息：用户决策、关键结论、错误信息
    return message.role === 'user' ||
           message.content.includes('决定') ||
           message.content.includes('错误');
  }

  private migrateToMemory(message) {
    // 提取关键信息，结构化存储
    if (message.role === 'user') {
      this.sessionMemory.addKeyDecision({
        id: message.id,
        description: message.content.substring(0, 100),
        timestamp: message.timestamp,
      });
    }
  }
}
```

## 6. 系统性风险总结

### 6.1 数据完整性风险 (P0)

**风险**: 文件锁与任务调度不协调 → 数据竞争 → 文件损坏

**触发条件**:
1. 用户要求并行任务
2. 两个SubTask修改同一文件
3. TaskDependencyGraph 未检测到冲突

**后果**:
- 文件内容被错误覆盖
- 快照无法正确还原
- 用户数据丢失

**修复优先级**: P0 - 立即修复

### 6.2 用户体验风险 (P1)

**风险1**: 消息重复 → UI混乱
**风险2**: 状态不同步 → 进度显示错误
**风险3**: 长会话上下文丢失 → 答非所问

**修复优先级**: P1 - 本周内修复

### 6.3 性能风险 (P2)

**风险**: 所有请求都初始化完整上下文 → 不必要的延迟

**场景**: 用户问"什么是TypeScript?" → 也创建快照管理器、会话、任务上下文

**影响**: 300ms延迟 → 用户感觉慢

**修复优先级**: P2 - 下周修复
