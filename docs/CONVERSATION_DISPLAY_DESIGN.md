# 对话内容展示设计方案

## 一、硬性要求

1. **不使用任何 emoji** - 全部使用现有 SVG 图标
2. **使用现有配色系统** - 不引入新色系
3. **简洁专业** - 避免花哨

## 二、现有配色系统（严格遵循）

```css
/* 角色颜色 */
--color-orchestrator: #8b5cf6;   /* 紫色 - 编排者 */
--color-claude: #c96442;          /* 橙褐色 - Claude */
--color-codex: #10a37f;           /* 绿色 - Codex */
--color-gemini: #4285f4;          /* 蓝色 - Gemini */

/* 状态颜色 */
--color-success: #22c55e;         /* 成功 */
--color-error: #ef4444;           /* 错误 */
--color-warning: #f59e0b;         /* 警告 */
--color-info: #3b82f6;            /* 信息 */
```

## 四、面板职责划分

### 核心原则

1. **角色区分** - 编排者(紫色) vs Worker(各自颜色)
2. **职责分离** - 主面板展示对话流，Worker面板展示执行详情
3. **交互优先** - 需要用户输入时，UI完全恢复到可输入状态

### 信息流架构

```text
用户输入
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│                     主对话面板 (Thread)                      │
│  - 用户消息                                                  │
│  - 编排者回复（含思考折叠）                                   │
│  - Worker 运行状态卡片（简要）                                │
│  - Worker 完成摘要（折叠）                                    │
│  - 最终总结                                                  │
└─────────────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │  Claude  │    │  Codex   │    │  Gemini  │
    │  面板    │    │  面板    │    │  面板    │
    └──────────┘    └──────────┘    └──────────┘
    各 Worker 的完整执行日志（任务卡片、思考、工具调用、输出）
```

## 五、主对话面板设计

### 5.1 定位

用户与编排者的对话界面。

**显示内容**：

- 用户输入消息
- 编排者回复（含思考折叠）
- Worker 运行状态卡片（简要：谁在做什么）
- Worker 完成摘要（折叠：结果概要）
- 交互请求（澄清/确认）
- 最终总结

**不显示**：Worker 的详细执行过程（工具调用、完整输出等）

**直接模式展示规则**：

- 用户直接选择 CLI 发送消息时，主面板不展示该 Worker 的详细输出
- 主面板不显示 Worker 状态卡片/摘要（仅编排模式才显示）
- 主面板仅保留编排者对话与最终总结内容

### 5.2 消息类型

| 消息类型    | 卡片变体            | 图标           | 说明                     |
| ----------- | ------------------- | -------------- | ------------------------ |
| 用户输入    | 无边框              | 无             | 用户消息                 |
| 编排者思考  | `card-orchestrator` | `orchestrator` | 默认折叠                 |
| 编排者回复  | `card-orchestrator` | `orchestrator` | 文本回复                 |
| Worker 状态 | `card-info`         | `info`         | 实时进度，含调度列表     |
| Worker 摘要 | 可折叠面板          | CLI 对应图标   | Worker 完成后显示        |
| 交互请求    | `card-warning`      | `question`     | 等待用户输入             |
| 完成总结    | `card-success`      | `success`      | 最终结果                 |
| 错误信息    | `card-error`        | `error`        | 执行失败                 |

### 5.3 布局结构

```text

│ 帮我重构 auth 模块，添加 JWT 支持                            │

## 六、Worker 面板设计

### 6.1 使用模式

Worker 面板有两种使用模式，显示内容不同：

| 模式         | 触发方式                         | 首条消息           |
| ------------ | -------------------------------- | ------------------ |
| **编排模式** | 编排者分配任务给 Worker          | 任务分配卡片       |
| **直接模式** | 用户直接选择 CLI 发送消息        | 用户消息           |

### 6.2 任务分配卡片设计（仅编排模式）

**视觉特征**：

- 使用紫色边框 (`variant: 'orchestrator'`)
- 图标使用 `orchestrator` 图标
- 标题为 "来自编排者的任务"
- 右侧显示执行者 CLI 名称

**内容结构**：

| 字段       | 说明                                       |
| ---------- | ------------------------------------------ |
| 任务描述   | 编排者分配的具体任务内容                   |
| 目标文件   | 需要操作的文件列表（如有）                 |
| 分配原因   | 为什么选择这个 Worker 执行（如有）         |
| 状态徽章   | 执行中 / 已完成 / 失败                     |

### 6.3 消息类型

| 消息类型       | 说明                                       |
| -------------- | ------------------------------------------ |
| 任务分配卡片   | 紫色边框，来自编排者的任务（仅编排模式）   |
| 思考过程       | 可折叠面板，默认折叠                       |
| 文本输出       | Markdown 渲染                              |
| 代码块         | 语法高亮，超过 20 行自动折叠               |
| 工具调用       | 时间线样式轨道                             |
| 完成状态       | 成功/失败样式                              |

### 6.4 代码块渲染规则

| 规则           | 说明                                       |
| -------------- | ------------------------------------------ |
| 语法高亮       | 使用 highlight.js，根据语言标识渲染        |
| 自动折叠       | 超过 20 行时默认折叠，显示前 5 行预览      |
| 文件名显示     | 代码块头部左侧显示语言/文件名              |
| 复制按钮       | 代码块头部右侧，点击复制全部内容           |
| diff 格式      | 绿色显示新增行，红色显示删除行             |

### 6.5 思考过程折叠规则

| 规则           | 说明                                       |
| -------------- | ------------------------------------------ |
| 默认状态       | 折叠                                       |
| 折叠时显示     | "思考过程" + 摘要（前 50 字符）            |
| 展开时显示     | 完整思考内容                               |

## 七、交互状态管理

### 7.1 状态机

```text
┌─────────┐    用户发送消息    ┌─────────────┐
│  Idle   │ ────────────────> │ Processing  │
│ (空闲)  │                   │  (处理中)   │
└────^────┘                   └──────┬──────┘
     │                               │
     │                               │
     │    ┌──────────────────────────┼──────────────┐
     │    │                          │              │
     │    v                          v              │
     │ 完成/取消              需要用户交互           │
     │                               │              │
     │                               v              │
     │                      ┌─────────────┐         │
     └───────────────────── │  Awaiting   │ <───────┘
                            │ (等待输入)  │
                            └─────────────┘
```

### 7.2 状态定义

| 状态           | 发送按钮             | 输入框               | 说明                 |
| -------------- | -------------------- | -------------------- | -------------------- |
| **Idle**       | 可点击，发送图标     | 可输入，默认占位符   | 初始状态             |
| **Processing** | 可点击，停止图标     | 禁用                 | 系统执行中，可中断   |
| **Awaiting**   | 可点击，发送图标     | 可输入，特殊占位符   | 等待用户输入         |

### 7.3 关键行为

当收到交互消息（需要用户澄清/确认）时：

1. **停止当前响应流程** - 暂停流式输出与进度更新
2. **立即停止处理状态** - `setProcessingState(false)`
3. **清除处理宽限期** - `localProcessingUntil = 0`
4. **恢复发送按钮** - 显示发送图标，可点击
5. **启用输入框** - 设置特殊占位符，可输入
6. **聚焦输入框** - 方便用户直接输入

## 八、错误与异常处理

### 8.1 错误场景

| 场景                 | 主面板显示                           | Worker面板显示         |
| -------------------- | ------------------------------------ | ---------------------- |
| 单个 Worker 失败     | 错误摘要卡片，显示失败原因           | 详细错误信息           |
| 编排者分解失败       | 错误卡片，提示重试或修改需求         | 无                     |
| 部分 Worker 失败     | 状态卡片显示各 Worker 状态           | 各自显示成功/失败      |
| Worker 超时          | 超时提示，可选择继续等待或中断       | 显示超时状态           |

### 8.2 错误卡片设计

- 使用 `card-error` 变体（红色边框）
- 图标使用 `error` 图标
- 显示错误原因和可能的解决建议
- 提供"重试"按钮（如适用）

**部分成功汇总规则**：

- 主面板仍生成最终总结卡片
- 总结中需同时展示成功项与失败项，并显式标记失败原因

## 九、Worker 完成摘要

### 9.1 摘要内容

| 字段         | 说明                                       |
| ------------ | ------------------------------------------ |
| 执行结果     | 成功 / 失败                                |
| 耗时         | 执行时长                                   |
| 修改文件     | 修改的文件列表（如有）                     |
| 关键输出     | Worker 输出的关键信息摘要（限 200 字符）   |

### 9.2 生成规则

- 由编排者根据 Worker 返回结果生成
- 每个 Worker 完成后生成一条摘要
- 摘要显示在主面板，默认折叠
- 点击可展开查看详情或跳转到对应 Worker 面板

## 十、消息更新策略

### 10.1 更新 vs 新建

| 消息类型             | 策略                                       |
| -------------------- | ------------------------------------------ |
| Worker 状态卡片      | 更新（同一任务只保留一个状态卡片）         |
| 流式消息             | 更新（根据 messageId 更新内容）            |
| 工具调用             | 追加（每个工具调用是独立条目）             |
| Worker 摘要          | 新建（每个 Worker 完成后新建一条）         |

### 10.2 去重规则

- 根据 `standardMessageId` 判断是否为同一消息
- 相同 ID 的消息更新而非新建
- 流式消息完成后合并为一条完整消息

## 十一、流式输出设计

### 11.1 流式阶段与完成阶段

| 阶段       | 渲染策略                                   | 说明                           |
| ---------- | ------------------------------------------ | ------------------------------ |
| 流式阶段   | 简单渲染，不解析复杂格式                   | 优先显示速度，容忍不完整格式   |
| 完成阶段   | 使用后端解析的 `parsedBlocks` 完整渲染     | 正确渲染 Markdown、代码块等    |

### 11.2 特殊格式处理

流式输出中可能包含不完整的特殊格式，需要特殊处理：

| 特殊格式           | 流式阶段处理                                    | 完成阶段处理                   |
| ------------------ | ----------------------------------------------- | ------------------------------ |
| 代码块 ` ``` `     | 检测未闭合的 ` ``` `，暂不渲染高亮              | 后端解析后正确渲染             |
| Markdown 加粗/斜体 | 原样显示，不解析                                | 使用 marked.js 渲染            |
| 思考标签           | 检测 `<thinking>` 开始，提取并折叠显示          | 后端解析后正确渲染             |
| 工具调用标签       | 检测 `<function_calls>` 等，不显示原文          | 后端解析后渲染工具调用轨道     |
| JSON 对象          | 检测裸露 JSON，暂不显示                         | 后端移除或转为代码块           |
| 链接/图片          | 原样显示 `[text](url)`                          | 使用 Markdown 渲染             |

**特殊格式包裹规则（流式阶段）**：

- 未闭合代码块、未闭合 JSON、未闭合思考标签等内容，统一包裹为“原始文本块”展示
- 原始文本块仅做基础转义与换行，不做高亮与 Markdown 解析
- 待消息完成后，使用完整解析结果替换原始文本块

### 11.3 流式渲染规则

```text
流式消息到达
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ 1. 检测特殊标签（thinking、function_calls 等）              │
│    - 有 <thinking> 开始标签 → 提取思考内容，折叠显示        │
│    - 有 <function_calls> → 隐藏，等待工具调用结果     │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. 过滤不完整格式                                           │
│    - 检测未闭合的 ``` → 暂时作为普通文本显示                │
│    - 检测未闭合的 ** 或 * → 原样显示                        │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. 简单渲染                                                 │
│    - 基本 HTML 转义                                         │
│    - 换行符转 <br>                                          │
│    - 不应用复杂 Markdown                                    │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. 显示流式指示器                                           │
│    - 消息底部显示 "思考中..." 或 "正在输出..."              │
│    - 显示已用时间                                           │
└─────────────────────────────────────────────────────────────┘
```

### 11.4 完成后重新渲染

消息完成（`lifecycle: 'completed'`）后：

1. **后端解析**：使用 `parseContentToBlocks()` 解析完整内容
2. **前端接收**：收到 `parsedBlocks` 数组
3. **完整渲染**：使用 `renderParsedBlocks()` 渲染
   - 代码块：highlight.js 高亮
   - Markdown：marked.js 渲染
   - 思考内容：折叠面板
   - 工具调用：时间线轨道

### 11.5 特殊标签检测正则（实现参考）

以下为实现参考，属于实现细节，可根据实际渲染逻辑调整：

```javascript
// 思考标签
const thinkingStartRegex = /<thinking>/i;

// 工具调用标签
