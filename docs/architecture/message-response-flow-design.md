## 1. 概述与问题分析

### 1.1 背景说明

本项目是一个 AI 编程助手，采用 **Orchestrator + Multi-Worker** 架构。用户通过对话界面发送任务，系统会经过分析、规划、执行等多个阶段完成任务。

**当前现状**：
- ✅ 消息卡片渲染已完善（文本、代码、工具调用、思考块等）
- ✅ 流式输出机制已建立
- ❌ 用户发送消息后的**即时反馈**体验不佳

### 1.2 问题诊断

通过分析当前交互流程，识别出以下核心问题：

#### 问题 1：视觉断档期

```
用户点击发送 ──────────────────────────────────────▶ 第一个响应出现
                    │                                      │
                    │◀──── 500ms ~ 3s 的空白期 ────────▶│
                    │                                      │
                    │  用户看到：                          │
                    │  - 用户消息已显示 ✓                  │
                    │  - 发送按钮变为停止按钮 ✓            │
                    │  - 消息区域：空白，无任何变化 ✗      │
```

**用户感受**：发送消息后看不到任何反馈，不确定系统是否在工作。

#### 问题 2：响应突然出现

当 AI 开始回复时，消息卡片是"突然弹出"的，没有视觉过渡，体验生硬。

#### 问题 3：状态不明确

用户无法区分以下状态：
- 消息正在发送到服务器
- 服务器已接收，正在排队
- AI 正在"思考"（规划阶段）
- AI 正在"生成"（输出阶段）

### 1.3 现有组件保持不变

| 组件 | 说明 |
|------|------|
| 消息卡片 (MessageItem) | 用户消息、助手消息的外观样式不变 |
| 内容渲染 (BlockRenderer) | 文本、代码、工具调用等渲染逻辑不变 |
| 工具卡片 (ToolCallBlock) | 工具调用的展开/折叠交互不变 |
| 思考块 (ThinkingBlock) | 思考过程展示不变 |
| 流式指示器 (StreamingIndicator) | 已有的三个跳动点可复用 |
| 顶部阶段步骤指示 | 不再采用，避免冗余信息 |

## 2. 设计目标与原则

### 2.1 核心目标

> **让用户在整个响应过程中始终感知到系统在工作，并明确当前的“响应状态”。**

### 2.2 设计原则

| 原则 | 说明 | 度量标准 |
|------|------|----------|
| **即时反馈** | 用户操作后必须有立即的视觉响应 | 反馈延迟 < 50ms |
| **状态透明** | 用户随时能理解系统当前在做什么 | 无需猜测当前状态 |
| **平滑过渡** | 状态切换时有自然的动画衔接 | 无视觉跳跃 |
| **最小干扰** | 动画不应分散用户对内容的注意力 | 动画时长 < 400ms |
| **增量改进** | 在现有稳定组件基础上优化 | 不重构现有渲染逻辑 |

### 2.3 成功标准

- [ ] 用户发送消息后 **50ms 内** 看到占位消息出现
- [ ] 用户能清晰区分 **"思考中"** 和 **"生成中"** 两个状态
- [ ] 从占位消息到真实内容的过渡 **无闪烁、无跳跃**
- [ ] 消息完成时有 **优雅的收尾动画**
- [ ] 支持 **prefers-reduced-motion**：仅保留“跳动点”作为最低限度的活动指示，其余装饰动画禁用

## 3. 消息响应生命周期

### 3.1 状态定义

| 状态 | 英文名 | 触发条件 | 持续时间 | 用户感知 |
|------|--------|----------|----------|----------|
| **等待中** | WAITING | 用户点击发送 | 0.5-3s | 占位消息 + 跳动点 |
| **生成中** | STREAMING | 收到第一个 token | 5-60s | 真实内容流式显示 |
| **已完成** | COMPLETED | 收到完成信号 | 终态 | 边框变绿，复制按钮显示 |
| **已失败/中断** | TERMINAL | 错误或用户取消 | 终态 | 错误提示或中断标记 |

### 3.2 状态转换图

```
                              用户发送消息
                                   │
                                   ▼
                            ┌───────────┐
                            │  等待中    │ ◀─── 用户消息 + 占位消息
                            │  WAITING  │      同时立即出现
                            └─────┬─────┘
                                  │
                    ┌─────────────┼─────────────┐
                    │             │             │
                    ▼             │             ▼
             ┌───────────┐       │      ┌───────────┐
             │  超时/错误  │       │      │  用户取消  │
             │  TERMINAL │       │      │  TERMINAL │
             └───────────┘       │      └───────────┘
                                 │
                                 ▼
                          ┌───────────┐
                          │  生成中    │ ◀─── 占位消息平滑替换为真实内容
                          │ STREAMING │◀────┐
                          └─────┬─────┘     │
                                │           │ 更多 token
                                └───────────┘
                                │
                    ┌───────────┼───────────┐
                    │           │           │
                    ▼           ▼           ▼
             ┌───────────┐ ┌───────────┐ ┌───────────┐
             │  已完成    │ │  已失败    │ │  已中断    │
             │ COMPLETED │ │  FAILED   │ │ CANCELLED │
             └───────────┘ └───────────┘ └───────────┘
                    ▲           ▲           ▲
                    └───────────┴───────────┘
                           TERMINAL 终态
```

### 3.3 时序流程

```
时间轴  0ms        100ms       1s         5s         10s
        │           │           │          │          │
用户    ──●发送──────────────────────────────────────────▶
视觉        │
        ─────●用户消息显示──────────────────────────────▶
            │
        ──────●占位消息(等待中)─────────────────────────▶
                     合并：发送中+已接收+思考中
                              │
        ───────────────────────●真实消息(生成中)────────▶
                                         淡入过渡
                                               │
        ─────────────────────────────────────────●完成──▶
```

### 3.4 设计合理性说明

| 说明项 | 结论 |
|--------|------|
| 状态数量 | 4 个足够覆盖用户可感知变化 |
| 可感知差异 | WAITING/STREAMING/COMPLETED/TERMINAL |
| 实现复杂度 | 低，路径单一 |
| 状态转换判断 | 简单、可测试 |

**关键洞察**：用户真正能感知的只有 4 个状态变化点：
1. 发送后立即看到反馈（WAITING）
2. 内容开始流式显示（STREAMING）
3. 内容完成（COMPLETED）
4. 出错或中断（TERMINAL）

## 4. 各阶段交互设计

### 4.1 阶段一：用户发送消息（WAITING 开始）

#### 4.1.1 触发条件

- 用户点击发送按钮
- 用户按下 `Cmd/Ctrl + Enter`

#### 4.1.2 即时响应（< 50ms）

| 元素 | 变化 |
|------|------|
| **输入框** | 内容清空，恢复默认 placeholder |
| **发送按钮** | 变为停止按钮 |
| **用户消息** | 立即出现在消息列表，带入场动画 |
| **占位消息** | 立即出现在用户消息下方 |

#### 4.1.3 用户消息动画

```text
动画参数：
- 类型：translateX(20px) + scale(0.98) → translateX(0) + scale(1)
- 时长：200ms
- 曲线：ease-out
```

#### 4.1.4 占位消息设计（强制）

> 本方案不使用骨架屏，改用简洁的跳动点 + 状态文字。原因：本项目消息结构复杂（工具卡片、思考块等），骨架屏无法准确预示内容结构。

```text
┌─────────────────────────────────────────┐
│  ┌───────────────────────────────────┐  │
│  │  ◉ Orchestrator    ●●●           │  │ ◀── StreamingIndicator
│  │  ────────────────────             │  │
│  │                                   │  │
│  │  ●  ●  ●   正在准备...            │  │ ◀── 复用 StreamingIndicator
│  │                                   │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘

样式：
- 左边框：主色（primary），表示活跃状态
- 背景：与普通助手消息一致
- 入场动画：fadeIn 200ms + translateY(8px)
- 卡片有微弱脉冲效果（box-shadow 呼吸）
```

**状态文字变化（强制且可落地）**：

| placeholderState | 显示文字 | 触发条件 |
|------------------|----------|----------|
| pending | 正在准备... | 发送后立即 |
| received | 已接收... | 后端确认接收 |
| thinking | 正在思考... | 编排进入分析 |

> 注意：上述状态文字变化为**强制要求**，由 MessageHub 统一下发 `placeholderState`。  
> 前端**只负责渲染**，禁止用时间轴或本地逻辑自行切换状态。

### 4.2 阶段二：开始流式生成（STREAMING）

#### 4.2.1 触发条件

收到第一个内容 token（`unifiedMessage` 或 `unifiedUpdate`）

#### 4.2.2 关键交互：占位消息 → 真实消息

这是最关键的过渡点。需要实现**原地更新**，避免：
- ❌ 删除占位消息 + 添加新消息（会导致闪烁）
- ❌ 滚动位置跳动
- ❌ 布局重排

**技术方案**：

```typescript
// 原地更新占位消息，而不是删除+新增
updateThreadMessage(placeholderMessageId, {
  id: realMessageId,        // 更新为真实 ID
  content: realContent,     // 填充真实内容
  isPlaceholder: false,     // 标记不再是占位
  wasPlaceholder: true,     // 标记曾是占位（用于过渡动画）
  isStreaming: true,        // 标记流式中
});
```

```text
过渡动画：
- 占位内容：fadeOut 150ms
- 真实内容：fadeIn 150ms
- 保持消息卡片边框和位置不变
- 无布局跳动
```

#### 4.2.3 流式生成状态

```text
┌─────────────────────────────────────────┐
│  ┌───────────────────────────────────┐  │
│  │  ◉ Orchestrator                   │  │
│  │  ────────────────────  12:34 PM  │  │
│  │                                   │  │
│  │  好的，我来帮你创建一个 React    │  │
│  │  登录组件。                       │  │
│  │                                   │  │
│  │  首先，我们需要安装必要的依赖：   │  │
│  │  npm install formik yup▋          │  │ ◀── 闪烁光标
│  │                                   │  │
│  └───────────────────────────────────┘  │
│    ▲                                    │
│    └── 边框高亮（主色）                  │
└─────────────────────────────────────────┘

视觉特征：
- 消息卡片边框：高亮主色
- 消息头部显示 StreamingIndicator（三个跳动点，强制）
- 复制按钮隐藏（流式中不可复制）
```

## 5. 视觉规范与动画设计

### 5.1 占位消息组件规范

#### 5.1.1 整体结构

```text
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   ┌─ 左边框（主色）                                     │
│   │                                                     │
│   │  ┌─ 头部 ─────────────────────────────────────────┐│
│   │  │  ◉ Agent 徽章                                  ││
│   │  └────────────────────────────────────────────────┘│
│   │                                                     │
│   │  ┌─ 内容区 ───────────────────────────────────────┐│
│   │  │  ●  ●  ●   正在准备...                         ││
│   │  └────────────────────────────────────────────────┘│
│   │                                                     │
└───┴─────────────────────────────────────────────────────┘
```

#### 5.1.2 状态对应的视觉元素

| 状态 | 左边框颜色 | 内容区 | 卡片效果 |
|------|-----------|--------|----------|
| **WAITING** | `--primary` | 跳动点 + 状态文字 | 微弱脉冲 |
| **STREAMING** | `--primary` | 真实内容 | 边框高亮 |
| **COMPLETED** | `--border`（默认） | 完整内容 | 无 |
| **TERMINAL** | `--destructive`（失败）或 `--muted`（中断） | 错误信息或已有内容 | 无 |

### 5.2 动画参数表（核心动画）

| 动画名称 | 类型 | 时长 | 触发时机 |
|----------|------|------|----------|
| **fadeSlideIn** | opacity + translateY | 200ms | 消息入场（用户/占位） |
| **dotPulse** | translateY + opacity | 1.4s 循环 | 等待状态跳动点 |
| **contentTransition** | opacity | 150ms | 占位→真实内容过渡 |
| **borderFade** | border-color | 300ms | 完成时边框变化 |

> 注：不允许任何“可选增强”或兼容路径。动画范围以本文档为唯一标准。

### 5.3 无障碍支持

```css
@media (prefers-reduced-motion: reduce) {
  /* 禁用装饰性动画 */
  .placeholder-message { animation: none !important; }
  .status-text { animation: none !important; }

  /* 保留最低限度活动指示：跳动点 */
  .dot-pulse { animation: dotPulse 1.4s ease-in-out infinite; }

  /* 边框颜色变化 → 即时切换（无过渡）*/
  /* 入场动画 → 即时显示 */
}
```

## 6. 边缘情况处理

### 6.0 稳定性强制规范（UX 保障）

> 以下规则用于保证用户体验稳定、响应明确，**不得省略**。

#### 6.0.1 首 token 超时处理（强制）

- 用户发送后 **8 秒**内必须进入 STREAMING 或 TERMINAL  
- 超过 8 秒仍未收到首 token → **直接进入 TERMINAL**  
- 必须显示明确原因：`等待响应超时，请重试`

#### 6.0.2 主消息流唯一绑定（强制）

- 主对话区只接收 **MessageHub 统一通道**的消息  
- 每次用户请求只绑定一个 `requestId`  
- 只有该 `requestId` 的首 token 可触发占位 → 实体替换  
- 其他 worker 的输出只能进入 **子任务视图**，禁止污染主消息流

#### 6.0.3 终态收口（强制）

- 进入 TERMINAL 后，所有动画必须立即停止  
- `StreamingIndicator`、占位状态、流式标记必须被清理  

### 6.1 网络异常

| 场景 | 处理方式 |
|------|----------|
| 发送请求失败 | 占位消息显示错误状态，提供重试按钮 |
| 长时间无响应 (>30s) | 自动超时，显示"请求超时"，提供重试 |
| 流式中断 | 保留已有内容，标记"连接中断" |
| 重连成功 | 继续从断点接收内容 |

### 6.2 快速操作

| 场景 | 处理方式 |
|------|----------|
| 快速连续发送 | 防抖处理，500ms 内忽略重复点击 |
| 发送后立即取消 | 由 MessageHub 统一下发取消与占位移除 |
| 流式中发送新消息 | 中断当前任务，开始新任务 |

### 6.3 会话切换

| 场景 | 处理方式 |
|------|----------|
| 切换会话时有进行中任务 | 后台继续处理，切回时同步状态 |
| 新会话加载历史 | 历史消息无占位状态，直接完整显示 |

### 6.4 长内容处理

| 场景 | 处理方式 |
|------|----------|
| 流式输出超长 | 自动滚动跟随最新内容 |
| 用户手动滚动查看历史 | 暂停自动滚动，显示"回到底部"按钮 |
| 代码块超长 | 代码块内部滚动，不影响整体布局 |

## 7. 参考对标分析

### 7.1 ChatGPT

| 特性 | 实现方式 | 借鉴价值 |
|------|----------|----------|
| 即时占位 | "ChatGPT is typing..." 占位 | ⭐⭐⭐ 核心参考 |
| 思考状态 | 无明确区分 | ⭐ 可增强 |
| 流式光标 | 黑色方块闪烁 | ⭐⭐ 可参考 |
| 完成过渡 | 无明显过渡 | - |

### 7.2 Claude.ai

| 特性 | 实现方式 | 借鉴价值 |
|------|----------|----------|

### 9.1 已存在的关键实现

| 方案要点 | 当前代码位置 | 说明 |
|---|---|---|
| MessageHub 统一通道 | `src/orchestrator/core/message-hub.ts` | 后端消息汇聚统一入口 |
| 消息流解析与状态更新 | `src/ui/webview-svelte/src/lib/message-handler.ts` | 负责把 hub 消息映射为前端状态 |
| 消息卡片渲染 | `src/ui/webview-svelte/src/components/MessageItem.svelte` | 卡片渲染已稳定 |
| 占位消息组件 | `src/ui/webview-svelte/src/components/PlaceholderMessage.svelte` | 已具备占位 UI |
| 流式指示器 | `src/ui/webview-svelte/src/components/StreamingIndicator.svelte` | 三点跳动组件 |

### 9.2 与设计稿的差异与修正点

| 设计要求 | 当前状态 | 结论 |
|---|---|---|
| WAITING 占位在发送后 <50ms 出现 | 由 MessageHub 统一下发 | 需确保后端立即发送 user + placeholder |
| 首 token 触发占位 → 实体消息原地替换 | 现有替换逻辑存在，但路径需统一 | 需统一 `message-handler` 的替换入口 |
| STREAMING 时内容区显示跳动点 | 当前仅在无块内容时显示 | 符合（避免头部干扰） |
| 复制按钮仅在非流式时显示 | `MessageItem` 已处理 | 符合 |

### 9.3 必须落实的稳定性规则

1. **WAITING 占位必须由 MessageHub 统一下发，前端禁止本地创建**  
2. **首 token 一律触发占位原地替换，禁止删除+新增**  
3. **所有状态切换必须通过统一消息通道，不允许多路径并行写 UI**

## 10. 总结

本设计方案通过以下核心改进解决消息响应流的用户体验问题：

### 10.1 核心改进点

| 改进 | 解决的问题 | 用户价值 |
|------|-----------|----------|
| **即时占位消息** | 视觉断档期 | 发送后立即看到反馈 |
| **状态模型** | 状态过多难实现 | 4 个核心状态，易实现易理解 |
| **平滑过渡动画** | 响应突然出现 | 视觉连贯，无跳跃感 |

### 10.2 设计原则落实

- ✅ **即时反馈**：< 50ms 响应
- ✅ **状态透明**：4 个核心状态清晰可感知
- ✅ **平滑过渡**：状态切换有动画
- ✅ **最小干扰**：动画时长 ≤ 300ms
- ✅ **增量改进**：不修改现有渲染组件
