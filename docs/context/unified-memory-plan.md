# MultiCLI 统一上下文与记忆系统基线方案

> 版本: 3.0（长期基线）
> 更新时间: 2026-02-06
> 适用范围: Orchestrator + 全部 Worker（Claude/Codex/Gemini）

## 1. 文档定位

本文档是 MultiCLI 在“多模型编排 + 多 Worker 并行执行”场景下的统一上下文与记忆系统唯一基线。

本基线目标不是“最小可用”，而是“长期稳定演进”：

1. 单一实现路径，避免同能力多套逻辑并存。
2. 降低重复读取与重复推理成本。
3. 提升跨 Worker 一致性与可恢复能力。
4. 用门禁与 SLO 驱动持续优化，而不是经验式调参。

## 2. 核心结论

### 2.1 唯一架构结论

采用 `SSOT + 三层记忆 + 统一组装器 + 强制门禁`：

1. `SSOT`：统一事实源（唯一写入与读取中心）。
2. `三层记忆`：短窗记忆、任务共享记忆、项目长期记忆。
3. `统一组装`：全部 Prompt 由 `ContextAssembler` 组装。
4. `强制门禁`：每次任务收尾执行自动化校验。

### 2.2 禁止项（工程硬约束）

1. 禁止同一能力存在多种实现路径。
2. 禁止保留回退分支掩盖主链路问题。
3. 禁止补丁式修复代替根因修复。
4. 禁止绕过统一上下文组装器直拼 Prompt。

## 3. 问题与根因

### 3.1 当前核心问题

1. 多 Worker 会重复读取相同文件，读放大明显。
2. Worker 之间缺少稳定的可复用事实交换层。
3. 直接执行路径与 Todo 路径行为不完全一致。
4. 上下文预算虽有配置，但缺少强制门禁闭环。

### 3.2 根因归类

1. 事实写入与任务执行主链路耦合不完整。
2. 文件摘要缓存能力存在，但未在所有读取路径强制前置。
3. 缺少全链路可观测指标和统一失败判定标准。

## 4. 目标与非目标

### 4.1 目标

1. 将“重复文件读取率”压降到可控区间。
2. 让每个 Worker 在可控 token 预算内获得连续上下文。
3. 保障崩溃恢复后可基于结构化事实继续执行。
4. 为后续模型扩展提供稳定协议与数据模型。

### 4.2 非目标

1. 不做跨项目全局知识混合。
2. 不在共享层存储原始长对话。
3. 不引入第二套并行记忆系统。

## 5. 统一架构

### 5.1 三层记忆模型

1. `L0 短窗记忆（Agent Local Window）`
   每个 Agent 最近固定 5 轮对话，仅用于保持当前执行连续性。
2. `L1 任务共享记忆（Mission Shared Memory）`
   Mission 范围结构化事实，供编排者与所有 Worker 复用。
3. `L2 项目长期记忆（Project Long-term Memory）`
   跨 Mission 的稳定知识，由 MemoryDocument 与项目知识库承载。

### 5.2 唯一事实源

统一事实源由以下组件组成，并对外暴露统一接口：

1. `SharedContextPool`：承载 L1 任务共享事实。
2. `FileSummaryCache`：`filePath + fileHash` 维度摘要缓存。
3. `ContextAssembler`：按预算组装最终 Prompt 上下文。
4. `ContextManager`：统一编排入口与生命周期管理。

说明：对执行层而言，`ContextManager` 是唯一入口，禁止 worker 私自接多套上下文源。

## 6. 统一数据模型

### 6.1 共享事实类型（固定集合）

`SharedContextEntry.type` 仅允许以下类型：

1. `decision`
2. `contract`
3. `risk`
4. `constraint`
5. `file_summary`
6. `insight`

禁止新增同语义别名类型，避免检索与组装分叉。

### 6.2 必填字段（写入门槛）

每条共享事实必须包含：

1. `missionId`
2. `source`
3. `type`
4. `content`（摘要文本）
5. `tags`
6. `importance`
7. `createdAt`

`file_summary` 必须额外包含 `fileRefs(path, hash)`。

### 6.3 写入约束

1. 内容必须摘要化，不允许长原文直写。
2. 同 `filePath + fileHash` 不允许重复缓存记录。
3. 相似事实按阈值去重或合并来源。
4. Mission 完结必须执行 L1 清理。

## 7. 主链路标准流程

### 7.1 写入流程（唯一）

1. Worker/Orchestrator 产出结论。
2. 结论结构化为共享事实。
3. 经写入校验器通过后写入 `SharedContextPool`。
4. 若为文件理解结果，同步写 `FileSummaryCache`。
5. 关键决策同步写入长期记忆。

### 7.2 读取流程（唯一）

1. 执行前调用 `ContextAssembler` 组装上下文。
2. 按 Agent 标签与最小重要性筛选 L1 条目。
3. 合并 L0 最近 5 轮与 L2 稳定记忆。
4. 预算超限时按策略裁剪低价值条目。

### 7.3 文件读取流程（强制前置缓存）

1. 先计算 `fileHash`。
2. 查询 `FileSummaryCache`。
3. 命中时优先使用摘要。
4. 未命中时读取原文并异步生成摘要回写。

禁止“命中后仍强制全量读文件”的回退分支。

## 8. Prompt 组装规范

### 8.1 预算分配（默认）

1. `L0 短窗记忆`：40%
2. `L1 共享事实`：35%
3. `L2 项目长期记忆`：15%
4. `运行约束与契约增强`：10%

### 8.2 裁剪策略（唯一）

1. 固定保留最近 5 轮（L0）。
2. L1 按 `importance + recency + tag match` 排序。
3. 超预算仅裁剪低分 L1 条目。
4. 不允许切换到“拼接原始历史”的替代策略。

### 8.3 路径一致性要求

1. `Todo` 执行路径必须使用同一组装器。
2. `Direct` 执行路径必须使用同一组装器。
3. 任一路径缺失共享上下文注入都视为未达标。

## 9. 治理与门禁

### 9.1 任务级自动化门禁

每次任务结束必须校验：

1. 是否写入至少一条可复用共享事实。
2. 文件读取是否经过缓存前置逻辑。
3. 上下文组装是否在预算阈值内。
4. 契约输入输出是否闭合。

任一失败即任务判定失败，禁止“忽略告警继续通过”。

### 9.2 指标与 SLO

核心指标：

1. 重复文件读取率
2. 缓存命中率
3. 单任务输入 token
4. 共享事实复用率
5. 契约闭合率
6. 中断恢复成功率

季度 SLO（建议基线）：

1. 缓存命中率 >= 70%
2. 重复读取率 <= 25%
3. 平均输入 token 较现状下降 >= 30%
4. 契约闭合率 >= 95%

6. `src/orchestrator/worker/autonomous-worker.ts`

### 10.2 必须补齐的主链路点

1. `Direct` 路径与 `Todo` 路径统一注入共享上下文。
2. `readFileWithCache` 强制接入实际文件读取主流程。
3. `writeInsight` 在任务收尾阶段强制执行。
4. `getRecentTurnsForAssembler` 按 `agentId` 实现隔离。
5. 门禁校验接入执行结果判定。

## 11. 分阶段实施路线

### 11.1 Phase A（架构冻结）

1. 冻结共享事实 schema 与写入校验规则。
2. 冻结组装预算与裁剪策略。
3. 删除并行旧策略入口。

### 11.2 Phase B（主链路收敛）

1. 统一 Direct/Todo 两条执行路径上下文注入。
2. 将缓存读取接入全部文件理解链路。
3. 打通事实写入与任务收尾闭环。

### 11.3 Phase C（门禁上线）

1. 接入任务级门禁校验器。
2. 接入关键指标采集。
3. 将门禁失败接入编排器失败流。

### 11.4 Phase D（SLO 驱动优化）

1. 依据指标调优权重与召回策略。
2. 定期清理低价值事实与过期缓存。
3. 扩展更多 Worker 时保持协议不变。

## 12. 风险与应对

1. 风险：共享事实噪声过高导致上下文污染。
   应对：提高写入阈值，强化去重与重要性判定。
2. 风险：缓存陈旧导致错误理解。
   应对：严格依赖 `fileHash`，变更即失效。
3. 风险：门禁过严造成吞吐下降。
   应对：先灰度统计，再按 SLO 分阶段收紧阈值。

## 13. 验收标准

架构验收必须同时满足：

1. Direct/Todo 路径上下文注入一致。
2. 文件读取全部经过缓存前置逻辑。
3. 任务收尾完成共享事实写入与门禁校验。
4. Mission 结束后共享上下文清理生效。
5. 指标链路可观测且可追溯。

未满足任一条，视为方案未完成落地。
