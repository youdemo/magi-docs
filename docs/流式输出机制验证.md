# Worker和Orchestrator流式输出验证

## Worker流式输出机制 ✅

### 代码流程 (adapter-factory.ts:151-193)

**真实流式处理**:
```typescript
// Step 1: 监听adapter的output事件 (每个chunk触发)
adapter.on('output', (chunk: string) => {
  const normalizer = this.getOrCreateNormalizer(type, source);
  let messageId = this.activeMessageIds.get(scopeKey);

  if (!messageId) {
    // Step 2: 首次chunk - 开始流式消息
    const traceId = this.generateTraceId();
    messageId = normalizer.startStream(traceId, source);  // → STARTED
    this.activeMessageIds.set(scopeKey, messageId);
  }

  // Step 3: 处理每个chunk - 发送流式更新
  normalizer.processChunk(messageId, chunk);  // → STREAMING updates
});

// Step 4: 消息完成 - 结束流式消息
adapter.on('response', (response: CLIResponse) => {
  const messageId = this.activeMessageIds.get(scopeKey);
  if (messageId) {
    normalizer.endStream(messageId, response.error);  // → COMPLETED
    this.activeMessageIds.delete(scopeKey);
  }
});
```

### 流式特性验证

✅ **真实逐字流式**:
- Worker每输出一个chunk → adapter.emit('output', chunk)
- Normalizer接收chunk → 解析并emit STREAMING更新
- UI接收更新 → 逐字显示内容

✅ **消息生命周期**:
1. 首个chunk → STARTED消息
2. 后续chunks → STREAMING更新(可能多次)
3. response完成 → COMPLETED消息

✅ **支持的流式内容**:
- 文本内容 → 逐字显示
- 代码块 → 逐行显示
- 工具调用 → 实时更新
- 思考过程 → 流式展示

```typescript
private emitOrchestratorMessage(type: CLIType, message: string): void {
  const summary = this.summarizeOrchestratorMessage(message);
  const normalizer = this.getOrCreateNormalizer(type, 'orchestrator');
  const traceId = `orchestrator-${Date.now()}-${Math.random().toString(36).substring(2, 9)}`;

  // 一次性发送完整消息,但走流式生命周期
  const messageId = normalizer.startStream(traceId, 'orchestrator');  // → STARTED
  normalizer.processChunk(messageId, summary);                         // → STREAMING
  normalizer.endStream(messageId);                                     // → COMPLETED
}
```

### 流式特性分析

✅ **标准流式生命周期**:
- startStream() → 发送STARTED消息
- processChunk() → Normalizer解析并发送STREAMING更新
- endStream() → 发送COMPLETED消息

⚠️ **内容一次性发送**:
- summary已经是完整的摘要文本
- 不是逐字生成,而是一次性传递
- Normalizer仍会解析并分块(如果内容有代码块/工具调用等)

✅ **设计合理性**:
- Orchestrator消息是**摘要**,本身就很短
- 一次性发送完整摘要是合理的
- 重要的是保持**标准流式协议**,而不是内容的逐字生成

## 验证结论

### ✅ Worker流式输出 - 完全正确

**真实流式处理**:
- 每个chunk都通过adapter事件流式发送
- Normalizer逐步解析并发送STREAMING更新

- 对于短摘要消息,这是合理且高效的设计
- 保持了协议统一性,UI可统一处理

**设计权衡**:
- 如果需要真正逐字流式,可以分块发送summary
- 但对于Orchestrator的任务摘要场景,当前设计已足够

## 可能的优化方向

### 选项1: Orchestrator真实流式 (不推荐)

```typescript
// 分块发送summary实现真正逐字流式
private emitOrchestratorMessage(type: CLIType, message: string): void {
  const summary = this.summarizeOrchestratorMessage(message);
  const normalizer = this.getOrCreateNormalizer(type, 'orchestrator');
  const traceId = `orchestrator-${Date.now()}-${Math.random().toString(36).substring(2, 9)}`;

  const messageId = normalizer.startStream(traceId, 'orchestrator');

  // 模拟逐字发送 (每次发送一个单词)
  const words = summary.split(' ');
  for (const word of words) {
    normalizer.processChunk(messageId, word + ' ');

  }

  normalizer.endStream(messageId);
```

- 用户期望快速看到任务摘要,而不是打字机效果

- 协议统一: 都使用StandardMessage
- 性能优化: Orchestrator消息快速显示,不阻塞
