## 实际风险评估

### 触发场景

**场景1**: 用户要求并行执行两个任务，修改同一文件
```
用户: "同时优化 types.ts 的类型定义和性能"

编排者生成计划:
- SubTask-1: 优化类型定义 (targetFiles: ['src/types.ts'])
- SubTask-2: 优化性能 (targetFiles: ['src/types.ts'])
- executionMode: parallel

实际执行:
T0: task-1 和 task-2 同时启动
T1: task-1 读取 types.ts (原始)
T2: task-2 读取 types.ts (原始)
T3: task-1 写入 types.ts (版本A)
T4: task-2 写入 types.ts (版本B) → 覆盖版本A ❌
```

**数据损坏风险**: 极高

### 快照系统的"保护"能力

**快照创建时机**:
```typescript
// snapshot-manager.ts:44-49
createSnapshot(filePath, modifiedBy, subTaskId) {
  // ... 读取当前文件内容
  if (fs.existsSync(absolutePath)) {
    originalContent = fs.readFileSync(absolutePath, 'utf-8');
  }
  // ... 创建快照
}
```

**并发执行下的快照**:
```
T0: task-1 createSnapshot('types.ts') → 快照A = 原始内容 ✅
T1: task-2 createSnapshot('types.ts') → 快照B = 原始内容 ✅
     (如果在 task-1 修改前)

或

T0: task-1 createSnapshot('types.ts') → 快照A = 原始内容 ✅
T2: task-1 修改 types.ts → 版本A
T3: task-2 createSnapshot('types.ts') → 快照B = 版本A ❌
     (如果在 task-1 修改后)
```

**快照保护失效的情况**:
- 如果 task-2 在 task-1 修改文件**后**创建快照
- 快照B 保存的是 task-1 的修改，而非原始文件
- 回滚 task-2 会丢失 task-1 的修改

## 实际触发概率评估

### 低概率因素

1. **用户很少要求并行执行修改同一文件的任务**
   - 大多数用户会自然地描述串行任务
   - 编排者也倾向于生成串行计划

2. **编排者的任务分解通常避免文件冲突**
   - 不同 SubTask 倾向于修改不同文件
   - 同一文件的修改通常在一个 SubTask 中

3. **快照系统的时间窗口很短**
   - 如果 task-2 在 task-1 修改前创建快照，问题不会出现
   - 只有在修改期间创建快照才会失效

### 高影响因素

1. **一旦触发，数据损坏严重**
   - 文件内容被错误覆盖
   - 快照无法正确还原
   - 用户可能丢失代码

2. **难以调试和复现**
   - 并发问题难以稳定复现
   - 用户很难理解为什么代码丢失

**综合评估**: 触发概率 5-10%，但影响极其严重

## 修复方案

### 方案1: 立即修复（推荐）

```typescript
// src/orchestrator/worker-pool.ts

async executeWithDependencyGraph(taskId, subTasks, context) {
  const graph = new TaskDependencyGraph();

  // 1️⃣ 修复：传递 targetFiles
  for (const subTask of subTasks) {
    graph.addTask(
      subTask.id,
      subTask.description,
      subTask,
      subTask.targetFiles || []  // ✅ 传递 targetFiles
    );
  }

  // 添加显式依赖
  for (const subTask of subTasks) {
    if (subTask.dependencies && subTask.dependencies.length > 0) {
      graph.addDependencies(subTask.id, subTask.dependencies);
    }
  }

  // 2️⃣ 新增：自动添加文件依赖
  const addedDeps = graph.addFileDependencies('sequential');
  if (addedDeps > 0) {
    console.log(`[WorkerPool] 检测到文件冲突，自动添加 ${addedDeps} 个依赖关系`);
  }

  // 分析依赖图
  const analysis = graph.analyze();

  if (analysis.hasCycle) {
    console.error('[WorkerPool] 检测到循环依赖:', analysis.cycleNodes);
    throw new Error(`任务存在循环依赖: ${analysis.cycleNodes?.join(', ')}`);
  }

  // ... 继续执行
}
```

**测试验证**:
