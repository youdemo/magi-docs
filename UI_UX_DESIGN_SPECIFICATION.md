# MultiCLI 交互界面设计规范

## 📋 文档概述

**目标**: 为 MultiCLI 插件定义清晰、实用、一致的交互界面设计规范

**设计原则**:
- ✅ **实用性优先** - 功能清晰，信息层次分明
- ✅ **简洁明了** - 避免花哨，专注核心体验
- ✅ **信息完整** - 正确使用 Markdown 和代码高亮
- ✅ **状态清晰** - 用户始终知道系统在做什么

---

## 🎯 核心架构理解

### 消息流向

```
用户输入
    ↓
IntelligentOrchestrator (编排者)
    ↓
任务分解 → Assignment
    ↓
AutonomousWorker (Worker)
    ↓
CLI 执行 (Claude/Codex/Gemini)
    ↓
输出 → StandardMessage
    ↓
前端显示 (主对话区 + Worker 面板)
```

### 显示职责划分

#### 主对话区 (Thread 面板)
**职责**: 展示用户与编排者的交互，以及 Worker 的**简要状态**

**显示内容**:
1. ✅ 用户输入消息
2. ✅ 编排者的规划和总结
3. ✅ 编排者的思考过程（折叠卡片）
4. ✅ 编排者的询问（澄清、确认）
5. ✅ **Worker 执行状态卡片** (subTaskCard)
   - 当前在做什么
   - 执行进度
   - 文件变更列表
6. ✅ **Worker 完成总结卡片** (summaryCard)
   - 执行总结
   - 完成状态
   - 耗时统计

**不显示**:
- ❌ Worker 的详细执行过程
- ❌ Worker 的完整代码输出
- ❌ Worker 的工具调用细节

#### Worker CLI 面板 (Claude/Codex/Gemini)
**职责**: 展示 Worker 的**完整执行过程**

**显示内容**:
1. ✅ **编排者分配的任务**（需要特殊标识 ← 待实现）
2. ✅ Worker 的详细思考过程
3. ✅ Worker 的完整输出内容
4. ✅ Worker 的代码块（带语法高亮）
5. ✅ Worker 的工具调用
6. ✅ Worker 的错误信息

## 🎨 交互状态管理

### 1. 正在处理状态

**视觉表现**:
```
┌─────────────────────────────────────────┐
│ [Assistant] 12:38                         │
│ 正在分析代码结构...                       │
│                                          │
│ ⋯ 正在输出 用时 3s                        │
└─────────────────────────────────────────┘
```

**实现要点**:
- 显示动画点（⋯）
- 显示用时
- 发送按钮显示"停止"图标
- 输入框禁用

### 2. 等待用户输入状态

**视觉表现**:
```
┌─────────────────────────────────────────┐
│ ⏸️ 等待您的回复                          │
│ 请输入补充信息...                         │
└─────────────────────────────────────────┘
```

**实现要点**:
- **UI 完全停止**（不显示"正在处理"）
- 显示明显的"等待回复"横幅
- 输入框自动聚焦
- 发送按钮显示"发送"图标
- 清除所有"正在思考"动画

### 3. 状态转换规则

```
正在处理 → 等待用户输入
  ↓
  ✓ 停止所有动画
  ✓ 清除"正在处理"标志
  ✓ 更新发送按钮图标
  ✓ 聚焦输入框
  ✓ 显示"等待回复"横幅

等待用户输入 → 正在处理
  ↓
  ✓ 隐藏"等待回复"横幅
  ✓ 设置"正在处理"标志
  ✓ 更新发送按钮图标
  ✓ 显示"正在思考"动画
```

**关键实现**:
- 使用 `isWaitingForUserInput` 标志位
- 防止后续消息覆盖状态
- 统一的 `resetToWaitingForInput()` 函数

## 🎨 Markdown 和代码高亮规范

### 1. Markdown 渲染

**支持的语法**:
- ✅ 标题 (H1-H6)
- ✅ 列表（有序、无序）
- ✅ 加粗、斜体
- ✅ 代码块（带语法高亮）
- ✅ 行内代码
- ✅ 引用块
- ✅ 链接
- ✅ 表格

**渲染位置**:
- ✅ 编排者消息内容
- ✅ Worker 消息内容
- ✅ 思考过程内容
- ❌ 用户消息（纯文本）

**实现函数**: `renderMarkdown(content)`

### 2. 代码高亮

**支持的语言**:
- TypeScript / JavaScript
- Python
- Java
- C / C++
- Go
- Rust
- HTML / CSS
- JSON / YAML
- Bash / Shell
- SQL

**实现方式**:
```javascript
// 使用 highlight.js
const highlighted = hljs.highlightAuto(code, [language]).value;
```

**显示规则**:
- 代码块超过 20 行 → 默认折叠
- 显示语言标签
- 显示行号（可选）
- 支持复制按钮

### 3. 特殊内容处理

#### 文件路径列表
```
检测到连续多行以 / 开头的路径
→ 添加 .has-special-content 类
→ 添加滚动提示
```

#### 带行号的代码
```
检测到 "数字→" 格式
→ 识别为带行号的代码
→ 保持格式显示
```

## 🎨 折叠面板规范

### 1. 使用场景

**应该折叠**:
- ✅ 思考过程（超过 3 步）
- ✅ 工具调用（超过 2 个）
- ✅ 代码块（超过 20 行）
- ✅ 长文本内容（超过 500 字）

**不应该折叠**:
- ❌ 普通 AI 回复
- ❌ 统计信息
- ❌ 简短的解释

### 2. 折叠面板结构

**HTML 结构**:
```html
<div class="collapsible-panel" data-panel-id="panel-123">
  <div class="collapsible-header" onclick="togglePanel('panel-123')">
    <span class="collapsible-icon">▶</span>
    <span class="collapsible-title">思考过程</span>
    <span class="collapsible-badge">5 步</span>
  </div>
  <div class="collapsible-content">
    <div class="collapsible-body">
      <!-- 内容 -->
    </div>
  </div>
</div>
```

**交互行为**:
- 点击标题栏展开/折叠
- 展开时图标旋转 90°
- 平滑动画过渡
- 保存展开状态

## 📝 设计决策记录

### 决策 1: 主对话区不显示 Worker 详细输出

**原因**:
- 保持主对话区简洁
- 用户主要关注整体进度
- 详细输出在 Worker 面板查看

**实现**:
- 使用 subTaskCard 显示简要状态
- 使用 summaryCard 显示完成总结

### 决策 2: 交互时 UI 完全停止

**原因**:
- 用户反馈：应该像接收新消息一样
- 避免用户困惑（以为还在处理）
- 明确的状态转换

**实现**:
- `isWaitingForUserInput` 标志位
- `resetToWaitingForInput()` 统一函数
- 防止状态被覆盖

### 决策 3: 编排者任务需要特殊标识

**原因**:
- Worker 面板中任务来源不清晰
- 用户需要知道这是编排者分配的
- 与 Worker 自主输出区分

**实现**:
- 蓝色徽章 "→ 编排者任务"
- 显示职责和任务描述
- 可折叠显示详细信息
