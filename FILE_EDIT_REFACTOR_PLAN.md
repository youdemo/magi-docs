# Magi File Edit Tool 重构架构设计与实现标准 (LLM as Editor + File Lock)

## 1. 背景与痛点

目前 Magi 采用的 `file_edit` 工具基于精确的字符串替换（`old_str`/`new_str`）或基于行号范围（`start_line`/`end_line`）进行编辑。当系统中存在 Orchestrator 和多个 Worker 并发执行时，这种机制暴露出严重的弱点：
* **单点事实丢失 (Loss of Single Source of Truth)**：大模型基于之前看到的旧文件内容生成的编辑指令，在实际执行时，文件可能已经被其他 Agent 或进程（如 `launch_process` 触发的脚本）修改。
* **行号失效与冲突覆盖**：旧的行号或旧的上下文匹配经常会失效，导致无法匹配到目标位置而报错，甚至在错误的位置进行替换，造成代码破坏或冲突覆盖。

## 2. 设计目标

参考 Augment 等先进 Agent 架构中的文件编辑模式，引入 **LLM as Editor (大模型作为独立编辑代理)** 和 **File-level Mutex Lock (文件级互斥锁)**，旨在彻底解决并发编辑冲突和行号/上下文过时问题。

**核心原则**：
1. **意图驱动 (Intent-driven)**：大模型工具调用只负责传递“要在哪里改”和“要怎么改”的高层意图，不再传递严格的、易碎的行号和上下文。
2. **实时读取 (Real-time Read)**：编辑操作发生瞬间必须读取磁盘/编辑器的**最新**文件状态，保证所见即所得。
3. **互斥隔离 (Mutual Exclusion)**：任何对同一文件的编辑操作必须串行化，防止并行写入导致的脏数据覆盖。

## 3. 架构设计与重构方案

### 3.1. 工具参数重构 (Schema Redesign)
废弃 `file_edit` 中的严格行号参数，采用意图驱动的参数：
- `file_path`: 需要编辑的文件路径。
- `edit_summary`: 对本次修改的简短描述（1-2 句话），用于日志记录和审查。
- `detailed_edit_description`: 详细的修改说明，可以包含自然语言指令、特定需求，或者需要替换的具体代码片段（search/replace snippets，但不需要严格行号）。

*注意：现有的 `file_insert` 或 `file_bulk_edit` 也可以视情况整合进此架构，或后续逐步迁移。初期先聚焦于高频且最容易冲突的 `file_edit` 改造。*

### 3.2. 文件级互斥锁 (File-level Mutex Lock) 机制
在执行层（推荐在 `src/tools/file-executor.ts` 内部，或全局引入一个 `src/utils/mutex.ts`）实现一个基于文件绝对路径的细粒度异步锁（Queue/Mutex）。

**工作流**：
1. Agent A 调用 `file_edit("src/a.ts")`。
2. 系统请求 `LockManager.acquire("src/a.ts")`。
3. 若锁空闲，Agent A 获取锁；若被 Agent B 占用，Agent A `await` 等待。
4. 获取锁后，立即从磁盘读取 `src/a.ts` **最新**内容。
5. 执行编辑操作。
6. 写回磁盘。
7. `LockManager.release("src/a.ts")`，唤醒等待队列中的下一个请求。

### 3.3. 独立 LLM 编辑代理 (LLM as Editor)
底层 `FileExecutor` 在获取到最新文件内容和工具参数后，需要调用一个底层的专职 LLM（例如通过依赖注入传入的 LLM Client 或特定的轻量级内部函数）来真正产生差异或新文件。

**Prompt 机制 (示意)**：
```text
你是一个专业的代码合并助手。
这是文件的最新内容：
<file_content>
{最新代码}
</file_content>

用户的修改意图如下：
<edit_intent>
Summary: {edit_summary}
Details: {detailed_edit_description}
</edit_intent>

请分析意图并将修改准确地应用到文件中。请直接输出修改后的完整文件内容（或者统一使用 diff 格式，视哪种更稳定而定，由于目前主流大模型在全量输出和 diff 上各有优劣，如果全量输出可控则直接要求全量输出）。
```

### 3.4. 对于非 LLM 进程修改的容错
如果外部脚本（例如通过 `launch_process` 运行了 `npm run lint --fix`）修改了文件，由于它是进程级别的，我们的应用级内存锁无法拦截它。
**容错机制保障**：只要外部进程的修改是在我们获取锁**之前**完成的，我们就能在加锁后读到最新的（已 fix）的代码；如果是在我们拿到锁**之后**和写回**之前**发生的，可能会产生竞态条件。但在大多数合理的工作流中，Agent 是顺序/按逻辑发起任务的。由于我们现在是基于最新内容让 LLM 再生成一遍最终状态，因此大模型自动具备了一定的冲突调解和适配能力（如果外部修改与意图不冲突，LLM 可能天然兼容；如果冲突，LLM 也更能基于最新上下文决定如何改）。

## 4. 实施步骤

1. **Step 1: 实现工具库层的文件锁 `AsyncMutex`**。
   - 创建简单的队列/Promise-based 锁机制。
   - 验证锁的加锁、排队、释放（包括异常情况下的安全释放）。

2. **Step 2: 改造 `file-executor.ts` 中的参数 Schema**。
   - 更新 `getFileEditDefinition`。
   - 适配相关用到 `file_edit` 旧参数的测试用例和 Prompt 提示词（如果在 System Prompt 里面写死了例子，需要更新）。

3. **Step 3: 注入 LLM Client 给 FileExecutor 或提升执行逻辑**。
   - 由于 `FileExecutor` 作为一个通用工具箱目前没有直接绑定 LLM 的能力，考虑在它的构造函数/或特定的 `execute` 方法上下文中，传入一个可用的轻量级（如 Haiku）或主 LLM Client 回调，或者将这一步逻辑提取到 `ToolManager` / `WorkerAdapter` 中去拦截处理。
   - （推荐：在初始化 `FileExecutor` 时，注入一个 `llmProvider`，专门用来提供生成修改后代码的纯文本完成接口）。

4. **Step 4: 组装 Prompt 与解析结果写回**。
   - 获取锁 -> 读取最新文件 -> 调 LLM 生成最新代码 -> `vscode.workspace.fs.writeFile` 写入 -> 释放锁。

5. **Step 5: 异常处理与重试**。
   - 保证如果在调用 LLM 阶段或写回阶段发生异常，`finally` 块中一定会释放对应文件的锁。
   - 若模型未能按规定格式输出修改（如返回了 Markdown 代码块而我们需要原始内容），做基本的净化（如使用 `BaseNormalizer` 或简单的正则去背部）。

## 5. 验收标准与验证依据

为保证此次重构成功且不会引入退化问题，设定以下验收标准（可作为后续编写自动化测试或手动走查的依据）：

| 验收项 | 验证说明/场景 | 预期结果 |
| --- | --- | --- |
| **并发锁互斥** | 编写一个测试，模拟两个 Agent (或直接两个 Promise) 同时（毫秒级间隔）触发针对同一个文件 `test.ts` 的 `file_edit` 调用。A 在文件开头加 `A`, B 在文件末尾加 `B`。 | **A 与 B 的修改均应存在**于最终文件中。如果不加锁，后者往往会覆盖前者（因为它们都读取到了旧的同一份未修改状态然后覆盖写）。引入锁后，必定是一个先写入，另一个读取更新后的状态再写入。 |
| **行号解耦** | 在新版 `file_edit` 的参数中只传入文件路径和详细意图（例如“把函数 X 的返回类型改成 Promise”），**不**传入行号。 | Agent 应该能利用后端的 LLM 编辑代理，精准找到函数 X 并修改成功，无需指定第几行。 |
| **异常恢复** | 故意模拟在调用 LLM 生成修改的过程中抛出超时或网络异常。 | 该文件的锁必须被正常释放。紧接着下一次（或别的 Agent）对该文件的编辑请求应该能立即获得锁并正常工作，不能出现死锁。 |
| **上下文感知容错** | 修改某文件：先调用 `file_edit` 将文件某部分的名称修改掉，然后在文件尚未完成写入前或之后立即再调用一次去修改紧邻的其他逻辑。 | 由于基于最新文件实时读取和再次让大模型去思考 Diff，即使第二步的大模型（发起调用的）基于的是非常老旧的记忆，底层编辑模型看到的是最新的文件，它仍能理解意图并在最新的代码形态上完成修改。 |
| **大文件性能** | 针对一个超过 2000 行的大文件发送局部修改的 `edit_summary` 意图。 | 底层编辑 LLM 应能正确处理并产生修改后的结果。这里需要验证 Prompt 设计和上下文窗口是否能承载整个大文件，以及大文件完全替换或使用 Diff 的解析准确率。 |
